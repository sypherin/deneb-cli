<ranking_spec>
<!-- Exact deterministic ranking for Task 1 so REC-03 tests can pin it. Prose only. -->

USE_CASES = ["coding", "vision", "chat", "general"]. Eligibility for a use_case: keep every catalog
model whose capabilities include use_case OR "general" (general models are always eligible as a
fallback; a vision use_case still requires the "vision" tag OR "general").

Per eligible model, pick its BEST FITTING quant: among quants where fits(model, quant, profile).fits
is True, choose the one with the HIGHEST bpw (best quality that still fits). If no quant fits, the
model is a non-fitting candidate (used only by the nothing-fits fallback below).

Rank the fitting candidates by this key, DESCENDING, with a stable deterministic tie-break:
  1. specificity: 1 if use_case in model.capabilities (specialized), else 0 (eligible only via "general").
  2. runs_acceptably: 1 if expected_speed_tier != "very-slow", else 0 (demote giants that "fit" but crawl).
  3. capability_proxy: params_b (MoE uses TOTAL params here — total ~ capability; active drove speed).
  4. speed_rank: index of expected_speed_tier in TIER_ORDER (fast highest).
  5. headroom_mb: more comfortable fit first.
  6. quant bpw: higher-quality quant first.
  7. model.name ASCENDING as the final stable tie-break (sort by name asc first, then apply the
     descending keys above so equal-key candidates stay name-ordered).
Return the top_n (default 3) as Recommendation records.

Recommendation dataclass: model: Model, quant: Quant, fit: FitResult, why: str.

why-string (pure, deterministic, no LLM) — one line built from the fields, e.g.:
  "<name> (<quant>): <role phrase> that fits your ~<usable_gb> GB budget with ~<headroom_gb> GB to
   spare; expected speed: <tier> on <backend>.[ <first caveat if any>]"
  where role phrase = "most capable <use_case> model" for the top pick, "<use_case>-capable" or
  "general-purpose" otherwise. Keep it plain English, NO em-dashes (house style).

NOTHING-FITS (REC-03): if the fitting-candidate list is empty, return a SINGLE Recommendation for
the smallest option: across the eligible set (or the whole CATALOG if the eligible set is empty),
pick the model+quant with the smallest fit.need_mb; its FitResult has fits False; why explains the
box is under-spec for this use_case, names the smallest option + how many MB short it is, and
suggests a smaller model / more RAM / CPU offload. Never return an empty list; never raise.
</ranking_spec>
