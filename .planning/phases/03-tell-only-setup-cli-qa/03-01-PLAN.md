<objective>
Build the pure, deterministic core of the tell-only setup advisory: `deneb/setup_advisor.py` -
a curated platform-branched PLAYBOOK data module + `setup_steps(model, profile) -> list[Step]`
that, given a catalog model and a Phase-1 HardwareProfile, produces the ORDERED setup steps for
THIS platform as ADVICE. Each Step carries the exact command, a one-line "what it does", and an
inline warning on anything that needs sudo / downloads N GB / starts a service (SET-03).

This is the heart of THE DENEB RULE for v1: Deneb TELLS the steps and NEVER executes them. The
module is PURE (string/struct in, Step structs out - no I/O, no tools import, no subprocess, no
LLM), so every generated step and every warning is unit-testable against constructed profiles
without touching the box (QA-01). Plan 03-02 wires this into `deneb setup <model>` and adds the
formal no-execution QA gate.

Purpose: answer "how do I actually set up this model on my box?" deterministically, platform-
correct, risk-flagged - the advisory half of the value, executing nothing.
Output: deneb/setup_advisor.py (SET-01/02/03) + tests/test_setup_advisor.py (QA-01).
</objective>

<setup_advisor_spec>
<!-- Exact contract for setup_advisor.py so the QA-01 tests can pin it. Prose only - NO fenced
     code blocks in the module (advice is directive text + curated command strings in DATA). -->

MODULE HONESTY CAVEAT (docstring, mirror models_catalog's): the PLAYBOOK commands are CURATED
BEST-EFFORT advice, not machine-verified for every box; the Deneb Rule means Deneb only TELLS
them - the user (or the paid executor tier, later) runs them. Verify before running.

Step dataclass:
  title: str            # short label, e.g. "Build llama.cpp (ROCm)"
  command: str          # the exact command to show (may contain the model's gguf repo + quant)
  what: str             # one-line plain-English "what it does"
  warnings: list[str]   # each applicable risk, inline (default_factory=list; empty when none)
  kind: str             # "runtime" | "download" | "run" | "note"

PLAYBOOK (the maintainable DATA module): a dict keyed by backend ("cuda","rocm","metal","cpu"),
each entry a small record with: a human label (e.g. "ROCm (AMD)"), the ORDERED runtime step
template(s) to stand up llama.cpp for that accelerator (a prebuilt/ollama note where that is the
sane path - Metal = brew), the backend-specific llama-server run flags (e.g. rocm/cuda "-ngl 999",
cpu = no -ngl), and any backend run-warning (e.g. the gfx1151/Strix kernel caveat on rocm). Keep
each runtime step as a template with {placeholders} filled by setup_steps - do NOT hand-inline
per-model commands in the playbook. An UNKNOWN/empty backend maps to the "cpu" (generic) entry
plus a caveat.

setup_steps(model, profile, quant=None) -> list[Step]  (PURE; guarded; NEVER raises):
  1. Determine backend = profile.primary_backend or "cpu"; if it is not one of the four known
     keys, use "cpu" and prepend a NOTE step: "could not read your platform confidently -
     showing generic CPU-safe steps; verify the accelerator flags for your box." (PLAT-03)
  2. Determine the advised quant: if quant is given use it; else pick the HIGHEST-bpw quant whose
     fits(model, quant, profile).fits is True (best quality that fits - same rule recommend uses).
     If NONE fit (under-spec / None budget), pick the LOWEST-bpw quant and add a warning on the
     download+run steps: "may exceed your box's memory budget - expect tight fit / OOM."
  3. Emit RUNTIME step(s) from PLAYBOOK[backend].runtime, filled. Any step whose command contains
     "sudo" (or installs system packages / ROCm) MUST carry a sudo warning: "needs sudo - installs
     system packages / drivers." A build step notes it compiles (kind="runtime").
  4. Emit the DOWNLOAD step (kind="download"): an `hf download <model.gguf_repo> --include
     "*<quant.name>*.gguf" --local-dir ~/models/<repo-basename>` style command, `what` = "downloads
     the <quant> GGUF weights", warning MUST name the size: "downloads ~<size_gb> GB" (size_gb =
     round(quant.size_mb/1024)). If model is a VISION model (has "vision" in capabilities, or notes
     mention mmproj), emit an extra download NOTE step for the mmproj projector GGUF (from the same
     repo) with `what` explaining vision needs the projector alongside the weights.
  5. Emit the RUN step (kind="run"): `llama-server -m ~/models/<repo-basename>/<file> <backend-run-
     flags> --host 127.0.0.1 --port 8001`, `what` = "starts the model server locally". Warning MUST
     flag the service: "starts a local server on 127.0.0.1:8001 - stays running until you stop it."
     On rocm with a gfx1151 GPU (gpu.extra.gfx in {gfx1151,gfx1150}), ALSO add the Strix kernel
     caveat warning. Bind 127.0.0.1 (never 0.0.0.0) - exposure is out of scope for v1 (paid tier).
  6. Return the ordered list. The whole body is guarded (try/except -> a single honest NOTE step
     "could not build setup steps for this model/box") so it NEVER raises (PLAT-03).

resolve_model(name) -> Model | None  (PURE):
  Match `name` against CATALOG case- and separator-insensitively (lower(), strip, drop spaces /
  '-' / '.' / '_' for the comparison key) so "qwen3.6-35b-a3b" resolves "Qwen3.6-35B-A3B". Return
  the Model on a match, None on no match. Never raises. (The CLI, Plan 03-02, turns None into a
  helpful error + a `deneb recommend` pointer.)

NO em-dashes in any generated `what`/`warnings` copy (house style - use a spaced hyphen).
</setup_advisor_spec>
